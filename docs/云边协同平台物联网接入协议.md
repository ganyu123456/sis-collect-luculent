**云边协同平台物联网软件接入****通信方案**

## ***\*文档目的\****

本方案基于KubeEdge云边协同架构设计，面向第三方网关软件厂商，明确平台通用MQTT Mapper的对接规范、MQTT三大主题（status/cmd/data）的消息格式及交互逻辑。第三方厂商仅需按本方案完成网关软件的MQTT通信适配与数据/状态格式化，即可通过平台通用MQTT Mapper实现标准化接入，无需适配平台底层KubeEdge DMI、Device Twin等能力，大幅降低适配成本。

本方案中平台通用MQTT Mapper为核心桥接组件，实现**平台侧设备模型/实例**与**厂商侧网关软件**的双向通信，完成状态同步、指令转发、业务数据透传的全流程能力。

## ***\*一、接入整体架构与核心逻辑\****

### ***\*1.1 核心架构\****

本接入方案基于**KubeEdge云边协同+通用MQTT Mapper**构建，核心分为**平台侧**、**Mapper桥接层**、**厂商网关软件侧**三层，架构逻辑如下：

1. **云端****平台侧**：指基于云边协同技术的物联网平台。完成设备模型定义、设备实例（Device Instance）创建与配置（含MQTT连接、主题、转发参数），通过KubeEdge Device Twin实现网关状态的孪生存储与管控指令下发；

**2.** ***\*边缘基础设施层\****：每个边缘节点***\*独立部署\****轻量化 MQTT Broker 服务（如 EMQX、Mosquitto），作为边缘节点统一消息枢纽，为网关软件、MQTT Mapper 提供标准 MQTT 消息中转能力；

3. **边缘****Mapper桥接层**：平台预开发的通用MQTT Mapper，按Device Instance配置的参数，与网关软件内置MQTT Broker建立连接，实现status主题状态采集、cmd主题指令转发、data主题业务数据透传，同时对接KubeEdge DMI接口完成设备孪生状态更新与指令同步；
4. **厂商网关软件侧**：完成工业现场真实设备的数据采集，以标准 MQTT 客户端身份连接边缘节点独立 MQTT Broker，按本方案规范将**网关自身状态**推送到status主题、**业务采集数据**推送到data主题，同时监听cmd主题接收平台下发的管控指令并完成响应。

1. **状态同步**：网关软件→边缘MQTT Broker (status主题)→MQTT Mapper采集→KubeEdge DMI→平台Device Twin（状态更新）；
2. **指令下发**：平台Device Twin（可写属性指令）→KubeEdge DMI→MQTT Mapper转发→边缘MQTT Broker (cmd主题)→网关软件监听并执行→边缘MQTT Broker (status主题)推送执行后状态数据；
3. **业务数据透传**：网关软件→边缘MQTT Broker( data主题)→MQTT Mapper采集→按Device Instance配置转发至指定MQTT目标地址/主题→平台数据面接收（不解析业务数据）。

### ***\*1.3\**** ***\*边缘\**** ***\*MQTT Broker 说明\****

边缘MQTT Broker为边缘节点统一部署的独立消息服务，由平台侧统一部署、运维、管理，无需网关厂商开发/集成，核心特性如下：

1. 部署方式：每个边缘节点仅部署 1 套轻量化 MQTT Broker 服务，支持集群 / 单机部署，兼容 EMQX、Mosquitto 等主流开源组件，默认监听标准端口（1883/TCP、8883/SSL）；
2. 核心作用：作为边缘统一消息枢纽，承接网关软件（MQTT 客户端）与平台 MQTT Mapper（MQTT客户端）的消息中转，实现状态、指令、业务数据的标准化流转；
3. 边界定义：边缘 MQTT Broker 是平台与网关软件的核心对接边界，网关厂商仅需以标准 MQTT 客户端身份连接该 Broker，即可完成全流程通信，无任何代码侵入、无额外组件依赖；
4. 安全保障：平台统一配置 MQTT Broker 的用户名 / 密码鉴权、SSL/TLS 加密，仅授权的网关软件、MQTT Mapper 可接入，保障边缘消息通信安全。

3. 通信一致性：网关软件与平台通用 MQTT Mapper 需使用相同的设备唯一编码与主题命名规则，确保消息能被正确识别与转发。

## ***\*五\*******\*、消息格式规范\****

网关软件与平台的所有MQTT交互均采用**UTF-8编码的JSON格式**，QoS级别统一为1（确保消息至少一次送达），以下为三大主题的消息格式详情。

### ***\*5\*******\*.1 状态主题（device/status）：网关\*******\*→\*******\*平台\****

#### ***\*5\*******\*.1.1 消息用途\****

网关按“***\*30秒周期+状态变更立即推送\****”的规则，向该主题推送全量状态数据，平台接收后更新网关状态展示。

#### ***\*5\*******\*.1.2 完整消息格式\****

{

 "timestamp": 1750453800000, // 消息发送时间（毫秒级Unix时间戳，必选）

 "runState": "Running",

 "collectPointTotal": 100,

 "collectPointOnline": 98,

 "lastCollectTime": 1750453800000,

 "collectInterval": 30,

}

#### ***\*5\*******\*.1.3 格式要求\****

1. timestamp为必选字段，用于平台识别消息时效性；
2. 所有字段名、数据类型需与平台确认的状态字段完全一致；
3. 每次推送需包含全量字段，禁止仅推送变更字段。

### ***\*5\*******\*.2 指令主题（device/cmd）：平台\*******\*→\*******\*网关\****

#### ***\*5\*******\*.2.1 消息用途\****

平台向该主题下发可写字段的修改指令（如调整采集间隔），网关监听并执行。

#### ***\*5\*******\*.2.2 完整消息格式\****

{

 "requestId": "CMD-20251020143000-001", // 指令唯一ID（平台生成，必选）

 "params": {

  "collectInterval": 60         // 指令参数：可写字段名+目标值（必选）

 },

 "timestamp": 1750453800000       // 指令下发时间（毫秒级Unix时间戳，必选）

}

#### ***\*5\*******\*.2.3 指令参数说明\****

· params内仅包含需修改的**可写字段名**和对应目标值，字段名需与“3.1 状态字段”中定义的可写字段完全一致；

· 单次指令仅支持修改一个字段（如需修改多个，平台会下发多条指令）；

#### ***\*5\*******\*.2.4 网关处理要求\****

1. 网关需实时监听该主题，指令到达后1秒内开始执行；
2. 网关识别params内的字段名（如collectInterval），按字段值修改对应配置，并驱动业务实现；
3. 执行完成后，需立即通过device/status主题推送更新后的全量状态（包含修改后的字段值），完成指令响应；
4. 指令执行失败时（如参数非法，采集间隔<1秒），需保持原有值，不在status主题中修改该字段，并在网关本地日志记录requestId和失败原因（如“collectInterval参数值0非法，需≥1”）。

### ***\*5\*******\*.3 数据主题（device/data）：网关\*******\*→\*******\*平台\****

#### ***\*5\*******\*.3.1 消息用途\****

网关将工业现场采集的业务数据推送到该主题，平台仅做透传存储，不解析数据内容。

#### ***\*5\*******\*.3.2 完整消息格式\****

{

 "timestamp": 1750453800000, // 数据采集时间（毫秒级Unix时间戳，必选）

 "gatewayId": "gw-industry-001", // 网关唯一标识（平台分配）

 "batchData": {        // 业务采集数据（必选，内部格式由厂商定义）

  "sensor_001_temp": 25.5,

  "sensor_001_hum": 60.2,

  "sensor_002_pressure": 1.01,

  "sensor_003_flow": 5.8

 }

}

#### ***\*5\*******\*.3.3 格式要求\****

1. timestamp/gatewayId/batchData为必选顶层字段，字段名不可修改；
2. batchData内的数据格式按照实际业务需求定义，在平台数据转发层面不解析此数据。

## ***\*六\*******\*、\*******\*技术规范\****

厂商仅需完成网关软件的***\*标准 MQTT 客户端适配\******、数据/状态格式化、业务采集逻辑实现**，无需关注平台底层KubeEdge、Mapper、MQTT Broker的实现细节，核心适配要求如下：

### ***\*6\*******\*.1 MQTT客户端适配\****

1. 网关软件需集成***\*标准 MQTT v5.0 客户端\****SDK（.NET：MQTTnet、Java：Eclipse Paho、Go：gmqtt、Python：paho-mqtt），无需内置 MQTT Broker；
2. 网关软件需集成**MQTT v5.0客户端**，实现以下能力：

· 发布：向status/data主题推送符合规范的JSON消息，QoS=1；

· 订阅：监听cmd主题，接收平台下发的指令，QoS=1；

· 重连：实现指数退避重连策略，断网后自动恢复MQTT连接；

3. 严格使用平台分配的边缘 MQTT Broker 地址、端口、鉴权信息，支持 SSL/TLS 加密连接。

### ***\*6\*******\*.2 状态上报适配\****

1. 严格按设备模型定义的属性名、数据类型、枚举值推送status主题数据；
2. 实现**周期上报+变更上报**的双重机制，确保平台能实时获取网关最新状态；
3. 网关启动/重启后，需在**5秒内**完成首次status主题全量状态推送。

### ***\*6\*******\*.3 指令处理适配\****

1. 网关软件需**实时监听**cmd主题，指令到达后**1秒内**开始执行；
2. 指令执行完成后，需立即通过status主题推送最新状态，实现指令响应；
3. 本地日志需记录所有指令的requestId、指令类型、入参、执行结果、执行时间，便于问题排查。

### ***\*6\*******\*.4 业务数据采集适配\****

1. 完成工业现场真实设备（DCS/PLC/传感器/设备/第三方服务）的业务数据采集，采集逻辑由厂商自主实现；
2. 按collectInterval配置的间隔推采集和推送data主题数据，batchData内的测点标识需确保业务可用**唯一且可追溯**；

### ***\*6\*******\*.5 异常处理与日志要求\****

1. 网关软件需实现完善的异常处理机制：MQTT连接异常、采集设备离线、指令执行失败等异常，需在本地日志中详细记录，并更新status主题的对应状态；
2. 日志需包含**时间戳、网关ID、事件类型、详细描述、关联ID（如requestId）**，日志保存周期不低于7天；
3. 网关软件需提供**健康检查接口**（如HTTP /health），返回网关运行状态、MQTT连接状态、采集任务状态，便于平台做容器化健康检测。

## ***\*七\*******\*、联调测试流程\****

平台侧与厂商侧按以下流程完成联调测试，所有测试项通过后，网关软件即可正式接入平台：

### ***\*7\*******\*.1 平台侧准备工作\****

1. 创建设备模型：基于网关能力定义statusTwins物模型属性；
2. 创建设备实例：配置configData（MQTT连接、主题、转发参数），关联设备模型与MQTT Mapper；
3. 部署边缘服务：在目标边缘节点部署独立 MQTT Broker，完成鉴权、加密配置；
4. 配置Mapper：启动平台通用MQTT Mapper，加载设备实例配置，连接边缘 MQTT Broker；
5. 准备测试工具：提供MQTT消息收发工具（如MQTT.fx）、平台Device Twin状态查看界面、指令下发界面。

### ***\*7\*******\*.2 厂商侧准备工作\****

1. 完成网关软件的MQTT适配，连接平台分配的边缘 MQTT Broker；
2. 完成状态上报、指令处理、业务数据采集的逻辑开发；
3. 部署网关软件至指定边缘节点，配置正确的 MQTT 连接参数；
4. 准备测试环境：搭建工业现场采集模拟环境，生成测试业务数据。

### ***\*7\*******\*.3 核心联调测试项\****

| ***\*测试项\**** | ***\*测试内容\****                    | ***\*验收标准\****                                           |
| ---------------- | ------------------------------------- | ------------------------------------------------------------ |
| MQTT连接测试     | 网关 / Mapper 与边缘 MQTT Broker 连接 | Mapper日志显示连接成功，无断连/重连异常                      |
